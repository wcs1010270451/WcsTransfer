# WcsTransfer 当前定版说明

更新时间：2026-04-22

本文档用于冻结当前版本能力，作为下一轮迭代的新会话起点。当前版本目标不是公开商用，而是：

- 部署到生产环境
- 配置 OpenAI 与 Anthropic 两类上游
- 进行少量、自测性质的稳定性观察

## 1. 当前版本定位

当前版本是一个多租户 LLM Gateway，包含：

- OpenAI-compatible 网关能力
- Anthropic 官方 Messages API 原生适配
- 管理后台
- 租户门户
- 钱包、计费、账单、对账、人工修账
- 生产部署与基础告警能力

不包含：

- 公开注册运营
- 自动支付/充值
- 多管理员体系
- 复杂 RBAC
- 完整外部监控平台集成

## 2. 已完成的核心能力

### 2.1 上游模型接入

已支持：

- OpenAI-compatible Provider
  - `POST /v1/chat/completions`
  - `POST /v1/embeddings`
  - `GET /v1/models`
- Anthropic 官方 Provider
  - `POST /v1/messages`

已验证的典型上游：

- OpenAI 官方
- 阿里百炼（OpenAI-compatible）
- Anthropic 官方 API

### 2.2 路由与可用性

已支持：

- 多 Provider
- 多 Provider Key
- 模型映射
- `fixed / failover / round_robin`
- Key 冷却与临时跳过
- 有限重试
- 流式透传
- 流式 usage 统计

### 2.3 管理端

管理端已支持：

- 管理员登录
- Provider 管理
- Provider Key 管理
- Model 管理
- 租户管理
- 租户用户管理
- Client Key 管理
- 日志中心
- 统计看板
- 调试页
- API Docs / OpenAPI / Swagger / ReDoc

当前管理员端已支持的关键租户动作：

- 新增租户
- 编辑租户
- 充值
- 人工账务修正
- 查看钱包流水
- 查看租户用户
- 创建租户用户
- 启用/停用租户用户
- 重置租户用户密码

### 2.4 租户端

租户端当前策略：

- 不允许自注册
- 只允许已有租户用户登录
- 租户用户由管理员创建

租户端已支持：

- 登录
- 查看可用模型
- 查看统计
- 查看日志
- 查看钱包流水
- 创建/禁用 client key
- 导出账单

### 2.5 鉴权与多租户

已支持：

- 管理员正式登录
- 租户用户登录
- 业务侧 `client_api_key`
- 模型授权
- 按租户隔离日志、统计、钱包、client key

### 2.6 钱包与计费

已支持：

- 租户钱包余额
- 管理员人工充值
- 管理员人工修账
- 模型双价格
  - 成本价
  - 对客售价
- 钱包按 `billable_amount` 扣费
- 请求日志记录：
  - `cost_amount`
  - `billable_amount`
  - `reserved_amount`
- 钱包流水结构化关联：
  - `trace_id`
  - `model_public_name`
  - `total_tokens`
  - `reserved_amount`
  - `cost_amount`
  - `billable_amount`

### 2.7 账务与对账

已支持：

- 钱包流水查看
- 管理端账单导出
- 租户端账单导出
- 定时账务对账
- 对账异常飞书/企微 webhook 告警
- 单租户异常只告警一次，恢复后再重新告警

### 2.8 生产加固

已完成的生产向能力：

- 生产环境默认关闭 docs
- 生产环境默认关闭 admin debug
- CORS 收口
- 基础安全头
- 管理端与门户禁缓存
- PostgreSQL 自动备份
- 恢复脚本
- 一次数据库恢复演练
- `/healthz` 外部 watcher
- 数据库/Redis 依赖告警
- Provider `429/5xx` 异常告警
- 钱包/预留金额拦截激增告警
- 账单扣费异常告警

## 3. 当前产品/流程策略

当前版本的运营逻辑是：

- 不开放用户自注册
- 不开放自动支付
- 不开放自动充值
- 不做多管理员
- 不做复杂角色权限
- 以单管理员 + 人工审核 + 人工充值 + 人工修账为主

这意味着当前版本更适合：

- 小规模内部试用
- 白名单试用
- 自测型生产部署

不适合：

- 公开放量
- 自动商业化运营
- 无人值守长期商用

## 4. 当前前端入口逻辑

当前前端行为：

- 根路由 `/` 直接进入 console
- 管理端未登录时会跳转 `/admin/login`
- 租户端登录页为 `/portal/login`
- 不再提供首页双入口
- 不再提供租户自注册

## 5. 当前需要人工配置的数据

生产环境部署后，至少需要人工配置这些初始数据：

- 1 个或多个 Provider
- 1 组或多组 Provider Key
- 至少 1 个启用的 Model
- Model 的价格与预留策略：
  - `cost_input_per_1m`
  - `cost_output_per_1m`
  - `sale_input_per_1m`
  - `sale_output_per_1m`
  - `reserve_multiplier`
  - `reserve_min_amount`
- 至少 1 个租户
- 至少 1 个租户用户
- 至少 1 笔租户钱包余额

## 6. 当前版本的已知取舍

以下是当前版本刻意保留的取舍，不算 bug：

- 只有一个管理员体系，不做管理员增删改
- 租户用户不能自助改密码
- 钱包修正为人工动作，不做自动修账
- 账务异常以人工排查和人工修正为主
- 监控以 webhook 告警为主，不做 Prometheus/Grafana 一体化
- 仍以 PostgreSQL + Redis + Docker Compose 为部署主形态

## 7. 当前版本仍然存在但不阻塞部署的事项

这些项目前未做完，但不阻塞当前“先部署跑几天”的目标：

- 容器退出/重启告警
- 更完整的外部监控平台接入
- 多管理员
- 用户自助改密码
- 自动支付/充值
- 邮件通知
- 更细粒度角色权限
- 更完整的前端国际化
- 更彻底的前端拆包优化

## 8. 当前版本建议的部署用途

建议用途：

- 部署到 GCP
- 配置 OpenAI 和 Anthropic 两类上游
- 用管理员创建少量租户和用户
- 观察：
  - 上游稳定性
  - 账单链路
  - 钱包扣费
  - 告警效果
  - Anthropic / OpenAI 两条链路是否稳定

不建议当前版本直接做：

- 公开注册链接
- 对外收款
- 大规模租户接入
- 正式 SLA 承诺

## 9. 下一版本迭代建议

如果下一轮要继续做，建议优先级如下：

1. 容器退出/重启告警

4. 文档与清单继续清理，去掉历史兼容描述
5. 更完整的监控平台接入
6. 更细的管理员审计展示页

不建议下一轮优先做：

- 支付
- 自注册
- 多管理员
- 复杂 RBAC
